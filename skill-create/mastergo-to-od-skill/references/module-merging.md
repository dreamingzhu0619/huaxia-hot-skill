# 模块同类合并判断规则

> 适用：Step 4 中判断多个原始模块是否应合并为一个组件类型。
> 原则：脚本做骨架签名比对给出候选，AI 从语义维度做最终确认。

---

## 核心问题

设计稿中可能出现多个结构相似、内容不同的模块（如"产品推荐理由 A""产品推荐理由 B""产品推荐理由 C"）。它们本质上是**同一种组件类型的不同实例**，应该合并为一套 assets（styles.css + decorations.html + template.html），用数组驱动多实例渲染。

判断的核心问题是：**这两个模块在设计意图上是不是同一类组件？**

---

## 三步排查法（从易到难）

### 第一层：看 FRAME 名称

脚本从 `_index.json` 和 `tree.md` 中读取每个模块的 `moduleName`（来自设计稿 FRAME 图层名）。

**信号（由强到弱）：**

| 信号 | 示例 | 置信度 |
|------|------|--------|
| 名称含字母后缀（A/B/C/拷贝/副本） | "产品推荐理由A" / "产品推荐理由B" / "产品推荐理由C" | 极高 |
| 名称共享相同前缀 | "推荐理由A" / "推荐理由B"，"产品卡1" / "产品卡2" | 高 |
| 名称语义同类 | "市场解读" vs "市场分析" | 中 |

**判定规则**：如果两个模块的名称含相同前缀 + 字母/数字后缀 → **直接判定为同类**，不需要继续下一层。

### 第二层：看正文内容

如果从名称上无法确定，读模块 JSON 中的 TEXT 节点内容，判断它们在**语义上是否属于同一主题**。

**判断维度：**

| 维度 | 问题 | 同类信号 | 非同类的信号 |
|------|------|----------|------------|
| 主题 | 两段正文在讲什么？ | 都在描述产品推荐理由 / 都在分析市场行情 | 一个在讲产品、一个在讲合规 |
| 结构 | 段落的组织结构是否一致？ | 都有"标题→正文→数据来源"结构 | 一个是列表、一个是连续段落 |
| 用词 | 关键词是否重叠？ | "红利""低波""指数"重复出现 | 关键词完全不重叠 |

**判定规则**：如果两段正文的语义主题相同（如都在描述"为什么推荐这个产品"），即使具体内容不同 → **判定为同类**。

### 第三层：比较固定元素与层级结构

如果名称和正文都无法确定，比较两个模块的**固定骨架**——即那些不随内容变化的部分。

**比较对象：**

1. **背景/底框 LAYER**：fill 颜色/渐变是否一致？borderRadius 是否一致？
2. **装饰 PATH/SVG**：是否有相同的装饰元素（分隔线、图标、角标）？
3. **节点 type 层级**：骨架签名是否一致？（diff_modules.py 做此事）

**脚本骨架签名比对规则：**

```
只看节点 type 层级，忽略：
  - 坐标值（relativeX/Y）
  - 文本内容（text）
  - 图表内部结构（图表 GROUP 标为 WILDCARD）
  - 节点名称（name）
```

**签名相同 → 脚本标记为候选合并组 → AI 确认。**

---

## AI 确认清单

对脚本给出的每个候选合并组，AI 需要确认：

- [ ] FRAME 名称是否含相同前缀 + 字母/数字后缀？
- [ ] 正文内容是否属于同一语义主题？
- [ ] 固定背景/底框的视觉属性是否一致（fill / borderRadius / border）？
- [ ] 骨架签名是否一致（忽略图表区域内部差异）？

**以上四项满足任意两项 → 确认合并。**

---

## 合并后的实例数量判定

合并确认解决了"是不是同一种组件"。但还需要回答第二个问题：**"下一期，这个组件会出现几次？"**

### 判定问题

> **换一期新的内容，这个组件可能增加或减少实例数量吗？**

### 三档结论

| 实例模式 | 含义 | content.template.json 写法 |
|----------|------|--------------------------|
| **固定1个** | 每期永远是1个，不会增减 | 单对象 `{ }` |
| **不固定（数组驱动）** | 每期数量可能不同（1~N个） | 数组 `[ { }, { } ]` |
| **不确定** | 单实例但不确定未来是否需要多份 → 保守用数组 | 数组 `[ { } ]` |

### 判定规则

#### 规则一：本期已出现多个 → 必然数组驱动

如果一个组件类型由多个原始模块合并而来（如推荐理由A+B+C = 3个），那么它必然是数组驱动。本期3个，下期可能是2个或5个。

#### 规则二：本期只出现1个 → 追问"它含产品/内容数据吗？"

| 特征 | 判定 | 原因 |
|------|------|------|
| 模块包含**产品名、产品代码、收益率**等产品数据 | **数组驱动** | 每期推荐的产品数量可能不同（GROWTH #1 教训：即使本期只有1个产品卡，下期可能推荐3个产品） |
| 模块包含**标题、正文、热点内容**等可变文本 | **看设计意图** | 如果设计师可能在一个页面里并列多个同类内容区块 → 数组驱动；如果明显是页面级唯一模块 → 固定1个 |
| 模块**纯装饰/品牌/合规**，0个 variable TEXT | **固定1个** | 银行logo、风险警告全文、品牌营销头图——页面级元素，天然只有1个 |
| 模块为**背景类**（底层背景图/底纹） | **固定1个** | 页面级元素 |

#### 规则三：不确定时 → 保守用数组

如果拿不准未来是否需要多份，默认用数组 `[ { } ]`。因为从数组改成单对象容易（去掉 `[ ]`），反过来要从单对象改成数组则需要改 template 结构。

### 典型判定表

| 模块 | 本期实例数 | 含产品/内容数据？ | 判定 |
|------|----------|-----------------|------|
| 产品推荐理由A/B/C | 3 | 是（标题+正文+图表） | **数组** |
| 产品卡 | 1 | 是（产品名+代码+收益率） | **数组**（下期可能多只产品） |
| 热点速递 | 1 | 是（正文+来源） | **看设计**（本期1个，如果设计意图允许多个热点并列→数组） |
| 营销头图 | 1 | 是（标题） | **固定1个**（页面级唯一头图，不会出现两个头图） |
| 银行结束语 | 1 | 否（纯PATH logo） | **固定1个** |
| 风险警告 | 1 | 否（合规条文+费用表格） | **固定1个**（页面级唯一合规区） |
| 背景图 | 1 | 否（纯LAYER背景） | **固定1个** |

### 多实例对 content.template.json 的影响

```
固定1个：
  "banner": {
    "titleLine1": "...",
    "titleLine2": "..."
  }

数组驱动：
  "recommendation": [
    { "mainTitle": "...", "body": "...", "chart": "...", "source": "..." },
    { "mainTitle": "...", "body": "...", "chart": "...", "source": "..." }
  ]
```

### 多实例对 template.html 和组装的影响

- template.html 只定义**一个实例**的 DOM 骨架
- 组装时 `build_html.py` 遍历数组，每个 `{}` 复制一份 template.html → 填充 → 拼接
- 增加一个实例 = 数组里加一个 `{}`
- 删除一个实例 = 数组里删一个 `{}`

---

## 合并后的产物对应

合并确认 + 实例数量判定后：

| 影响 | 处理方式 |
|------|----------|
| assets 数量 | 合并组只产生 1 套 assets |
| 实例数量 | 固定1个 → 单对象；数组驱动 → `[ { } ]` |
| styles.css | 从代表模块提取（通常是第一个实例） |
| decorations.html | 从代表模块提取固定装饰 |
| template.html | 只定义单实例骨架，组装时按数组长度复制 |
| 图表差异 | 不同实例的图表统一用 `{{chart}}` 插槽，不 diff 内部结构 |

## 示例：huaxia-hot-citc

**输入**：9 个原始模块

**脚本骨架签名比对结果**：全部标记为 singleton（因为推荐理由 B/C 的图表区域内部结构不同）

**AI 三步排查**：

| 模块 | 第一层（名称） | 第二层（正文） | 第三层（底框） | 结论 |
|------|-------------|-------------|-------------|------|
| 推荐理由A vs B vs C | "产品推荐理由A/B/C" → 含字母后缀 | 都在描述"为什么推荐红利低波" | 红色渐变底框 `#FF4E3C→#FF1818` 一致 | **合并** |
| 产品卡 vs 其他 | 名称无共同前缀 | 产品信息 vs 推荐理由 vs 热点分析 | 金色渐变 `#FCCC89→#EB8F2C` vs 红色 vs 蓝色 | **独立** |

**合并 + 实例数量判定后**：

```
原始模块              合并后组件            实例模式      content.template.json
───────────────────────────────────────────────────────────────────────────
背景图                → background         固定1个       { }
产品卡                → product-card       数组驱动      [ { } ]（产品数据，下期可能多个）
推荐理由A + B + C     → recommendation     数组驱动      [ { }, { }, { } ]
营销头图              → marketing-header   固定1个       { }（页面级唯一头图）
银行结束语            → bank-ending        固定1个       { }（纯PATH logo）
风险警告              → risk-warning       固定1个       { }（页面级唯一合规区）
热点速递              → hotspot-express    数组驱动      [ { } ]（本期1个，但设计意图允许并列多个热点）
```
