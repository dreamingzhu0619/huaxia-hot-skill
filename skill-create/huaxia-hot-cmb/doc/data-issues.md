# 数据常见疑问

记录从 MCP 原始数据到归一化过程中遇到的数据结构问题与解答。

---

## 1. PATH 节点为什么都没有子节点？

**现象**：整个设计稿 160 个 PATH 节点，全部零子节点（当前设计稿实测数据）。在 MasterGo 界面上，某些"形状结合" PATH 内部能看到"组 → 矩形"的嵌套层级，但 API 返回时都是叶子。

**原因**：MasterGo MCP 对 PATH 类型（包括 boolean union 形状结合、路径）做了展平处理。导出时多个形状已被合并成单条 `path.data`（SVG 路径字符串），原始的操作数结构不保留。

**影响**：层级树里 PATH 显示为叶子节点是**正确的**，不影响最终渲染。`path.data` 存的就是合并后的最终几何，画出来的效果和 MasterGo 一致。丢掉的只是设计师编辑用的中间结构，不是最终画面。

**对 fixed/variable 的意义**：PATH 本质上是固定装饰（蒙版、背景曲线、图标、表格线框），每期长图都一样，写在 `path.data` 里不变。真正会变的东西是 TEXT 节点（文字和数值），而不是去改 PATH 的长度。

**相关文件**：`data/raw/01-getDsl/getDsl.json` → 节点的 `path[].data` 字段。

---

## 2. 层级关系以什么为准？

getDsl 自带的 `children` 字段就是权威的父子关系，不需要额外推理或修正。设计师在 MasterGo 里怎么建的，`children` 就怎么反映。不要用边界框包含关系去"修正"层级——那会把设计师有意放在平级的节点错误地嵌套。

---

## 3. getDsl 和 section DSL 有什么区别？

- **getDsl**：完整节点树，包含全部样式和属性，是权威来源。
- **section DSL**（`getDesignSections` 分段）：精简版，PATH 的 svgHtml 被剥离、长文本被替换为占位符、坐标可能归零。只能作为补充，不能覆盖 getDsl 数据。

合并时原则：**先来的更全，全的不能被缺的覆盖**。

---

## 4. 文本占位符是什么？

getDsl 中超过 50 字的文本可能被替换为占位符 `T{n}|{nodeId}`（如 `T1|0:958`）。真实文本在 `data/raw/05-getDesignTexts/getDesignTexts.json` 里。归一化时会自动替换回去。

---

## 5. 节点类型速查：PATH / FRAME / GROUP / LAYER

| 类型 | 有子节点？ | 是什么 | 在本项目中的角色 |
|------|-----------|--------|----------------|
| **FRAME** | 有 | 画框/画布，有自己的坐标系，可裁切内容 | 页面根节点、banner、各个section容器 |
| **GROUP** | 有 | 绳子，把几个东西捆在一起移动 | 位图蒙版组、标题装饰组 |
| **LAYER** | 无 | 素材，图片/色块等 | 背景光效、位图填充 |
| **PATH** | **全部无** | 钢笔描的矢量形状，用数学公式描述 | 装饰曲线、蒙版、表格边框、图标、布尔运算结果 |

额外概念：

- **矢量**：用数学公式存图，放大不糊。PATH 就是矢量的。对面是**位图**（像素格子），放大就糊。
- **SVG**：一种矢量图文件格式，本质是 XML 文本，浏览器读到就能画出形状。`path.data` 存的就是 SVG 路径数据。
